# Tobacco Atlas Web 骨架屏与首屏性能优化设计

## 背景

当前 `tobacco-atlas-web` 已经开始迁移到 `Next.js App Router`，首页 `/`、品牌列表 `/brands`、制造商列表 `/manufacturers` 已接入 `src/app` 路由体系。但当前加载体验和首屏性能仍有明显问题：

- 首次进入页面时容易感知到白屏或长时间空白等待
- 首页在首屏阶段同时装配 Hero、产品网格和装饰性浮动产品，关键路径较重
- 路由切换时缺少 `App Router` 原生分段加载态
- 首页产品区直接使用大列表渲染，导致初次请求和首屏编译成本偏高

本次目标是通过骨架屏和首屏减载两条线并行优化，让用户更早看到稳定内容，同时实际减少首屏等待时间。

## 目标

- 减少 `/`、`/brands`、`/manufacturers` 的白屏感
- 为关键页面补充路由级和组件级骨架屏
- 降低首页首屏渲染负担
- 让装饰性内容不再阻塞关键内容
- 尽量利用 `Next.js App Router` 的加载机制而不是自建复杂状态机
- 在不重做视觉系统的前提下，提升加载体验和首屏速度

## 非目标

- 本轮不优化全部详情页
- 本轮不重写数据层来源
- 本轮不引入复杂的客户端虚拟列表
- 本轮不同时做大规模美术升级

## 当前问题分析

### 1. 路由级加载态缺失

虽然页面已经迁入 `App Router`，但还没有为已接管页面提供 `loading.tsx`。当服务端准备页面内容或首次编译页面时，用户容易看到整页空白。

### 2. 首页首屏关键路径过重

首页当前结构中：

- `Navbar`
- `HeroSection`
- `ProductGrid`
- `MobileNav`

都会在首屏请求阶段尽快参与渲染，而 `ProductGrid` 直接映射完整产品集合，`FloatingProducts` 也属于额外负担。对于首个请求或冷启动，这会拉长可见内容出现时间。

### 3. 装饰性内容优先级过高

`FloatingProducts` 是视觉增强模块，不是用户首屏完成任务所必须看到的内容。它应该让位于：

- 导航
- 站点标题与引导信息
- 第一屏可浏览的产品卡片

### 4. 骨架屏缺失导致等待不可预期

即使页面最终可用，用户在等待时没有任何结构性反馈，就会主观感知为“卡住”或“白屏”。

## 总体策略

采用“加载体验优化 + 首屏减载”同步推进：

1. 为已经进入 `App Router` 的关键页面增加路由级 `loading.tsx`
2. 为首页产品区提供组件级 skeleton
3. 把首页首屏数据量从“完整列表”降为“首批内容”
4. 将装饰性浮动产品从关键路径中降级
5. 保持 Server Component 负责页面壳和数据切片，Client Component 只保留必要交互

## 设计方案

### 方案 A：只加骨架屏

做法：

- 给页面补 `loading.tsx`
- 给产品区补 skeleton

优点：

- 改动最少
- 白屏感会明显减轻

缺点：

- 只改善感知，不一定缩短真实等待时间
- 如果首屏仍然很重，用户会看到“骨架停很久”

### 方案 B：只做性能优化

做法：

- 裁剪首页首屏数据
- 降级装饰性模块
- 减少客户端负担

优点：

- 能缩短真实等待时间

缺点：

- 路由切换和冷启动时仍可能出现空白等待
- 用户体验不够稳定

### 方案 C：骨架屏 + 首屏减载

做法：

- 路由级 skeleton
- 首页组件级 skeleton
- 首页首批内容裁剪
- 装饰性模块延后

优点：

- 同时改善“感知速度”和“真实速度”
- 适合当前迁移阶段，收益高且风险可控

缺点：

- 需要同时触达页面结构与部分组件边界

## 推荐方案

推荐采用方案 C。

理由：

- 当前问题不只是“用户看到了空白”，而是“首屏确实偏重”
- 只做其中一半都不够完整
- 方案 C 可以在不大动数据层的前提下获得最快收益

## 页面级设计

### 首页 `/`

#### 路由级加载态

新增 `src/app/loading.tsx` 或针对首页的加载组件，提供与首页结构相匹配的骨架：

- 顶部导航骨架
- Hero 文本块骨架
- 产品网格骨架
- 移动端底部导航预留

目标是让首屏在数据未就绪前仍保持稳定布局。

#### 产品区骨架

新增产品卡片骨架组件，例如：

- `ProductCardSkeleton`
- `ProductGridSkeleton`

骨架需保留真实卡片比例、网格间距和标题区位置，避免加载完成时产生明显跳动。

#### 首屏减载

首页不再在首屏阶段渲染完整产品集合，而是：

- 先取首批产品作为首屏集合
- 剩余产品延后显示，或在首屏外区域渲染

首批数量按双端布局设定，原则是：

- 移动端足够填满 1 到 2 屏
- 桌面端足够形成完整网格节奏
- 不为了“看起来很多”而一次性压入全部内容

#### 装饰模块降级

`FloatingProducts` 进行如下处理之一：

- 使用 `next/dynamic` 延迟加载
- 只在大屏显示且不参与关键路径
- 在首页骨架未完成前不渲染

推荐先使用延迟加载 + 大屏限定，避免其影响首屏关键资源。

### 品牌列表 `/brands`

#### 路由级加载态

新增 `src/app/brands/loading.tsx`，包含：

- 页头骨架
- 区域切换按钮骨架
- 搜索框骨架
- 品牌网格骨架

#### 性能优化

品牌页核心负担相对首页更轻，本轮重点以骨架体验为主，不强行做复杂数据裁剪。

### 制造商列表 `/manufacturers`

#### 路由级加载态

新增 `src/app/manufacturers/loading.tsx`，包含：

- 页面标题骨架
- 搜索与筛选条骨架
- 制造商卡片骨架

#### 性能优化

制造商页已经带有筛选和排序逻辑，本轮重点保持现有行为稳定，优先通过骨架和较轻的首屏装配减少等待感。

## 组件与文件设计

建议新增或修改如下文件：

```text
src/
  app/
    loading.tsx
    brands/loading.tsx
    manufacturers/loading.tsx
  components/
    skeletons/
      PageShellSkeleton.tsx
      HeroSkeleton.tsx
      ProductCardSkeleton.tsx
      ProductGridSkeleton.tsx
      BrandGridSkeleton.tsx
      ManufacturerGridSkeleton.tsx
  components/
    FloatingProducts.tsx
    ProductGrid.tsx
    ProductCard.tsx
```

说明：

- 将 skeleton 组件集中到一个小目录，避免散落在业务文件中
- 骨架组件尽量不带业务逻辑，只负责视觉占位
- 首页的 `ProductGrid` 应接收更明确的输入，例如首批产品列表，而不是默认总是完整映射

## 技术策略

### 1. 使用 App Router 原生 loading 机制

优先使用 `loading.tsx`，而不是自己维护复杂的本地 loading state。

好处：

- 与 Next.js 路由切换行为天然一致
- 更适合服务端页面
- 不需要额外的全局状态

### 2. 减少首屏客户端工作

遵循当前最佳实践：

- 页面壳尽量保持为 Server Component
- 仅交互性组件保留为 Client Component
- 避免把大列表和装饰性模块全都挤进首个客户端边界

### 3. 延迟非关键视觉内容

`FloatingProducts` 和其他纯视觉增强内容属于非关键内容，应在：

- 首屏核心内容出现之后
- 桌面端环境下
- 浏览器空闲或低优先级阶段

再逐步展示。

### 4. 优先骨架稳定，避免布局跳动

Skeleton 设计需提前占住真实布局位置，防止：

- 导航高度变化
- Hero 高度变化
- 网格卡片突然重排

## 风险与应对

### 风险 1：骨架和真实内容差异过大

应对：

- 骨架严格复用真实布局比例
- 先追求“结构相似”，再考虑视觉精致

### 风险 2：首页裁剪首批内容后视觉密度下降

应对：

- 让首批数量覆盖完整首屏
- 在首屏之外继续衔接更多内容

### 风险 3：延迟装饰模块影响视觉完整性

应对：

- 先保障功能和首屏可见内容
- 装饰效果以渐进增强方式出现

### 风险 4：Next 首次编译仍然偏慢

应对：

- 骨架屏确保等待期间不白屏
- 组件边界继续朝“更轻的首屏”调整

## 验收标准

优化完成后，应满足：

- `/` 首次进入时显示稳定骨架，而不是白屏
- `/brands` 和 `/manufacturers` 在路由切换或首次加载时有明确加载态
- 首页首屏不再一次性渲染完整产品集合
- `FloatingProducts` 不阻塞首页关键内容出现
- `typecheck`、相关测试和页面访问验证通过
- 页面加载后无明显布局跳动

## 推荐下一步

在该设计确认后，应输出一个简洁的实现计划，按顺序执行：

1. 新增 skeleton 组件与 `loading.tsx`
2. 改造首页产品区为“首屏批次 + 非关键延后”
3. 降级 `FloatingProducts`
4. 运行测试、类型检查和路由访问验证
